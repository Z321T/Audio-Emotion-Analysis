# Audio-Emotion-Analysis

基于 Qwen2-Audio + EmoLLM + Qwen3-TTS 的语音陪护链路项目，用于完成以下任务：

- 音频转写（中文）
- 情绪识别（输出整体情绪）
- 基于转写与情绪生成陪护回复（面向老人）
- 将回复文本合成为语音并自动播放
- 本地文件批量测试
- 麦克风语音输入输出闭环（自动录音、自动播报）

## 项目概述与核心链路

本项目实现了一个完整的“语音理解 -> 情感陪护 -> 语音合成”交互链路，具体流程如下：

1. **语音输入及理解 (ASR & Emotion)**
   - **模型**：`Qwen/Qwen2-Audio-7B-Instruct`
   - **逻辑**：分析输入的音频文件，返回结构化的分析结果（包含 `{"asr": "...", "emotion": "..."}`）。
2. **文本生成及情感陪护 (LLM Reply)**
   - **模型**：`aJupyter/EmoLLM_Qwen2-7B-Instruct_lora`
   - **逻辑**：结合用户的语言（ASR转写结果）和情绪特征，生成富有同理心的陪伴式文字回复。
3. **语音合成及播放 (TTS)**
   - **模型**：`Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`
   - **逻辑**：将大模型生成的回复文本合成为包含自然情感表现力的音频文件，完成整个业务闭环。

**显存优化机制**：
为确保该链路在消费级 GPU 上顺利运行，全链路代码（如 `demo_asr_llm_tts.py`）设计了显存管理和回收机制（`release_cuda_memory`）。阶段执行完毕后主动将当前已被使用的模型卸载并在 CUDA 侧进行清理，让不同模型阶段能够安全串行化并共享有限的显存资源，防止出现 OOM。

## 目录与模块说明

* **`audio_emotion/load_models/`**：模型下载与加载模块。
  * `download_model.py`：从 ModelScope 自动拉取所需模型快照，并将绝对路径写入本地 `.env` 环境变量文件。
  * `load.py`：基于 `.env` 配置的路径加载相关模型。
* **`utils/`**：各处理环节的核心业务功能封装，使用了 `asyncio` 异步执行和重载。
  * `audio_asr_analysis.py`：驱动 Qwen2-Audio 处理音频并输出结果。
  * `emotion_llm_reply.py`：驱动 EmoLLM 生成文本回复。
  * `audio_tts.py`：调用 Qwen3-TTS 生成并保存语音文件。
* **`demo_example.py`** 为贯通所有流程的综合样板。

## 快速开始

### 1. 环境配置

推荐使用 Python >= 3.10。项目基于 `pyproject.toml` 管理依赖。
可使用常规方式安装所需环境：
```bash
pip install -e .
```
推荐使用 uv 环境管理工具来安装依赖：
```bash
uv sync
```

> **提示**：为确保能够正常使用相关模型库，建议安装带 CUDA 支持的 PyTorch 版本。

### 2. 准备模型

运行下载脚本自动化获取相关的模型权重。执行完毕后，对应的模型路径会自动记录在当前项目的 `.env` 当中，无需手动配置：
```bash
uv run downloadmodels.py
```

### 3. 运行全链路体验

准备好环境与模型后，可以直接运行串联起从“输入音频”到“陪伴语音”的完整演示代码：

1. 将一段测试音频放在 `test_audio/` 目录下（例如默认提供或自行录制的音频文件）。
2. 控制台运行：
```bash
uv run demo_example.py
```
执行完毕后，终端将依次打印出转写与情绪分析结果、LLM的陪护文本回复内容，并提示合成出的最新语音文件的保存路径。
