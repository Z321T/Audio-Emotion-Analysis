# AGENTS.md - Coding Agent Guide

本文件用于约束在 Audio Emotion Analysis 仓库中工作的 AI 编码代理。

## 函数注释规范

新增或修改函数时，必须为函数编写统一风格的中文文档字符串，便于后续维护、联调和自动化理解。

要求如下：

- 函数注释统一使用三段式结构：`简要说明`、`参数`、`返回`
- 第一段先说明函数的主要功能，即“这个函数是做什么的”
- `参数` 部分不仅写参数名称，还要说明参数用途、类型语义和必要注意事项
- `返回` 部分必须说明返回值是什么；如果返回字典、列表、嵌套对象等复杂结构，必须继续写清字段含义
- 如果函数没有正常返回值，要明确写出“无”以及是否会直接抛出异常
- 与业务相关的文档字符串优先使用中文，不要只保留过于简短的一句话注释

推荐模板：

```python
async def example(arg1: str, arg2: int = 0) -> dict[str, str]:
    """
    简要说明这个函数完成的主要功能。

    参数：
        arg1: 参数 1 的含义、使用场景与注意事项
        arg2: 参数 2 的含义。若有默认值约束、可选条件或业务限制，也要写清楚

    返回：
        返回结果说明。
        如果返回复杂结构，需要继续说明字段，例如：
        {
            'field_a': '字段 A 的含义',
            'field_b': '字段 B 的含义',
        }
    """
```

如返回结果为列表嵌套对象，也应按字段说明其结构与取值语义，不要只写“返回评测结果”之类的笼统描述。

## 当前开发范式（2026-03）

以下范式用于约束后续模型对话与代码修改，保证项目在本地开发与服务器部署时行为一致。

### 一、整体链路约束

- 当前主链路固定为：用户语音输入 -> 音频转写与情绪识别 -> 陪护回复生成 -> TTS 语音合成与播放。
- 音频理解模型使用 Qwen2-Audio-7B-Instruct。
- 文本陪护模型使用 EmoLLM_Qwen2-7B-Instruct_lora。
- 语音合成模型使用 Qwen3-TTS-12Hz-1.7B-Base。
- 音频分析输出结构固定为：{"transcription":"...","emotion":"..."}。

### 二、12Hz 优先策略

- 项目当前默认仅支持 12Hz 路径，不再主动启用 25Hz 分词器。
- 新增功能时，若非明确需求，不得重新引入 25Hz 依赖路径。
- 修改 TTS 相关代码时，应优先兼容当前 transformers 版本，避免依赖特定私有 API。

### 三、模型加载与复用规范

- 三类模型必须采用单例缓存复用，避免在单轮会话中重复初始化。
- 服务启动阶段优先调用统一预热接口，减少首轮推理时延。
- 禁止在高频函数中直接重复 from_pretrained。

### 四、导入与包结构规范

- audio_emotion 包内模块调用统一使用相对导入。
- 对外暴露接口尽量收敛在包级入口，避免业务层直接耦合底层实现文件。
- 修改时保持向后兼容；若必须破坏兼容，需在文档中明确升级说明。

### 五、异步与流程控制规范

- 涉及 I/O 或重计算的流程优先使用异步封装。
- 端到端流程函数应提供同步兼容入口，便于脚本与服务两种场景调用。
- 对麦克风采集、推理、合成、播放应进行分段异常捕获，避免整链路一次失败后直接崩溃退出。

### 六、依赖管理规范

- 依赖变更必须同步更新 pyproject.toml，并保持最小必要依赖。
- 仅在功能必需时引入新依赖，优先减少重量级或平台强耦合依赖。
- 遇到第三方库版本不兼容时，优先做兼容层或回退策略，而非直接锁死整套环境。

### 七、验证与交付规范

- 每次关键改造后，至少完成：
    1) 静态错误检查。
    2) 模型初始化或预热级别的最小可运行验证。
    3) 关键路径日志可读性检查。
- 若无法在当前环境完成完整音频实测，必须在交付说明中明确未验证项与建议验证步骤。
